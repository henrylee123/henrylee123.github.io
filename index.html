buffer cache 的区别

cache

cache 主要作用缓解 cpu 与普通内存的速度差异。把频繁的寻址地址缓存在 cache 能够优化系统性能。

cache 有三级，一级比一级快。

buffer

buffer 一定程度上减少对 i/o 的设备访问次数。 比如，写入一个文件到磁盘中的时候，会先存一部分在 buffer 里，攒到一定量的时候再写回磁盘中。减少对此盘的访问次数

  2. free
  3. top (进程排序 MEM %)

概述

三者都是 cpu 程序执行的时间段

进程

一个程序任务的开始到结束。进程

进程的切换

CPU 运行速度太快了，只有寄存器能够跟上它的速度。RAM 与 其他各种资源都无法跟上他的步伐。
CPU 不可能干等着，为了保持它不断工作。所以有了进程的切换，进程们排着队，等着 cpu 临幸它们。

一般进程的切换步骤：保存当前进程的上下文，调入下一进程上下文（一些程序执行时用到缓存下来的值）。执行下一进程，

切换上下文的步骤有

  1. 切换内存上下文（页目录，地址空间）。
  2. 把 cpu 寄存器内容保存在栈里：
  • 例子

进程的切换好比老师们帮学生改作文，100 个学生排着队改，老师改作文一看就知道哪里有问题圈个红圈，指点几句，学生根据老师的指点自己回去改，改了之后再回来给老师看，改得不好的再改。

  • 没有进程的切换：老师要指点第 1 个学生的作文哪里有问题，等他重写问题部分，再检查，再指点，直到作文没有错误才轮到第 2 个学生。
  • 有进程的切换：老师指点完第 1 个学生作文，他到一边上去改，老师马上帮第二个学生看。第一个学生改完作文后台排在第 100 个学生后面排队等待下次作文检查。一直到所有学生作文没有错为止

线程

线程解决的问题：

接着上面的例子，有个学生非常笨拙。写的作文又长又臭，一个老师看一遍要看许久。一位老师觉得等我看完其他同学都毕业了。于是把他的文章剪成几段，分给其他的老师同时检查。
进程的任务量过大，分成几份给 cpu 运行，会加快改任务完成。

线程的切换

  • 讨论的线程为内核级线程

  • 上下文切换

    只需要保存寄存器的内容，不需要切换内存上下文。 该进程的内存线程可以共享。

  • 切换性能

    同一进程内的线程的切换比进程的切换开销要小。进程切换 cpu 高速缓存区的缓存内存地址全部清空，内存地址空间发生变化。线程的切换则不会。

协程

协程是用户级的线程

用户级线程 & 内核级线程

  • 用户空间、内核空间

    操作系统设计者规定的内核空间，用户空间是进程执行的时候内存分配给该进程的两种栈。操作系统的危险操作都会先从用户空间切换到内核空间（系统调用，中断等）。

tcb

线程表, 记录每个线程的程序计数器、堆栈指针、寄存器和状态

用户态与内核切换：

  • 从用户态切换到内核态的途径
  1. 中断(被动)
  2. 异常(被动)
  3. 陷入指令（访管指令）(主动)

指令系统分级 0-3 级 其中 0 级为最高级
只有进入内核态可以运行 0 级指令：如内存清零，设置时钟，停机等等

用户级线程：

由程序自己选择合适的时间切换线程，程序阻塞时的线程调度由程序实现。能切换到所在进程拥有自己的 tcb

  • 优点
    1. 创建销毁线程、线程切换的代价小一些，因为内核的进程调度算法会频繁切换运行进程
    2. 更可控，由程序控制切换，如果程序有问题在某个线程阻塞了，整个进程都会被挂起。
    3. 能够利用更多的堆栈空间
  • 缺点
    1. 只能利用单个 cpu 资源，因为内核里面没有存该进程的 tcb，所以内核并不知道是多线程，不能分配多核资源。

内核级线程（守护进程）：

内核调度线程。当有线程阻塞时，会自动切换。tcb 存在内核中。

  • 优点
    1. 可以利用多核资源
    2. 自动切换，调度代价比进程小
    3. 一个线程阻塞时内核会调用另一个线程
  • 缺点
    1. 频繁的用户态内核态切换，效率不如用户级线程

Linux常见的调度算法

发表于 2019-11-19

进程调度算法

进程一经批准就会进入就绪队列等待唤醒，唤醒后开始 cpu 为该进程服务分配好的时间长度，进程的调度发生在就绪队列里。

1.先来先服务

根据等待队列的执行顺序来执行任务，是一种非抢占式服务

  • 优：利于长作业及 cpu 繁忙的作业
  • 劣：不利于短作业及 io 繁忙的作业

2.短作业优先调度算法

根据对任务预先执行时间对作业进行预估，短作业可以分配到更优先的资源

  • 优：利于短作业
  • 劣：不利于长作业

3.时间片轮转法

作业等待时间越长，优先级更高。作业在 cpu 运行完一个时间片的时间后如未运行完则回到队列入口，等待时间置 0。

4.多级反馈队列算法

设置多个优先级队列，按优先级遵循以下规则：

1.优先级越高时间片越高, 时间片约

  • 优
    1.为提高系统的吞吐量和缩短平均周转时间而照顾短进程
    2.为获取较好的 i/o 设备利用率而缩短响应时间照顾 i/o 型进程
    3.不必顾忌进程的执行时间，动态调节

*.关键名词注解

时间片：一次作业调度的基本时间间隔

Flask 小结

发表于 2019-11-14

背景

最近失业了，有空学习并总结一下后端的知识点。用 flask 用的较多，所以随意写写 flask 的基础与拓展。

基础

flask 框架是灵活性高一个中小型的 web 框架。采用 MTC 模式。说是这样说，但是 Model 都不是 flask 内置的。本身只内置了 jinja 模板语法，与可以编写指定路由的页面展示逻辑的函数。

路由

路由中可以接受 int float 等类型

  • 已注册的路由可以由逻辑函数得到

路由并发

启动 app.run() 的时候不开 mutil thread 或者 mutil process

在一个路由函数里 time.sleep(1000) 另一个路由函数里 time.sleep(1)

请求 test1 时阻塞了，但却可以访问 test2。单线程单进程启动的 app 怎么可以在程序阻塞时运行别的路由函数。flask 里面没有实现，但是 flask 基于 Werkzeug, Werkzeug 里实现了用户级的线程调度来调度线程，避免被阻塞。

同时请求 test2 两次，两次返回的时间为 20 秒。判断第 2 次请求等待第一次请求完成才开始执行路由函数。所以，flask 开启的用户级线程数跟路由函数相关，同一路由函数不能同时应付两个请求。

启动

开发时可以用 app.run(debug=1) 来开启调试模式。

  • 在线上启动的时候默认可以选择
    • 默认（单线程）
    • 多线程 threaded=True
    • 多进程 processes=4

部署

Flask 自身实现的并发方法并不是最佳实践，故一般采用默认的启动模式。并发性能的问题交给 uwsgi 或 gunicorn 来并发启动 worker。

mysql 主从，与读写分离

读写分离

读写分离采用一主多从的方式，数据库的业务一般读多写少。读锁是共享锁，写锁是排它锁。大并发量下，对数据库会造成很大压力。

为什么要做读写分离

随着业务发展，数据量越来越多。度成为系统发展的瓶颈，这时候，应该先引入缓存，memcache，redis。原因是缓存的开发复杂度远远小于读写分离。

当缓存也成为了系统瓶颈的时候，我们考虑数据库分库分表，读写分离的操作。

方案

数据库 master 负责读写数据， 从机负责读数据

数据库主机通过 binlog 把数据同步给从机，每台服务器都有全量的数据。binlog 为 mysql 的二进制日志，用于记录用户对数据库操作的 sql 语句信息 可以使用 mysqlbin 命令查看二进制日志的内容。
