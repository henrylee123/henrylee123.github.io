树算法

Posted on 2019-12-02 Edited on 2019-12-05 In 算法

B 树 （B- 树）

一个 m 阶的 B 树定义如下：

  1. 每个节点最多有 m 个孩子
  2. 除根节点和叶子节点外，每个节点至少有 m/2 个孩子
  3. 每个叶子节点都在同一层
  4. 所有叶子节点都在同一层
  5. 关键词的格数 n 满足 ceil(m/2)-1 <= n <= m-1
  6. 右子树的键值 > 本节点键值 > 左节点键值
  • 插入操作
  1. 普通插入

  1. 插入后叶子节点元素超过阶数 m 分裂，中间元素塞入上一级节点

  1. 插入后页子节点元素超过阶数 m 分裂后，中间元素塞入上一级节点，上级节点元素超过阶数 m

B+ 树

B 树上的非叶子节点存储的数据量比较小，叶子节点存储的数据也太少了。
不大适合在磁盘上使用该索引算法。磁盘的特点是 io 操作比较慢，所以需要的树深度必须要小一些。
B+ 树为 B 树的一种改进，为了适应磁盘的索引算法，

与 B 树的不同在于：

  1. 非叶子节点只存键值，数据只存在叶子节点。因为这样做可以使非叶子节点能存下更多的键使树的深度变小
  2. 所有叶子节点奸都有一个链指针 （这里个人觉得也是为了适合磁盘的场景，有的数据可能就存在连续的几个磁盘块内，每次都从 root 开始太慢了）

Mysql 的 Innode 存储引擎索引中用的就是 B+ 树。一个页（非叶子节点）的大小可调控默认为 16 kb。一般一个 page 可以存 1k 个键值。三层的 B+ 树索引就可以存储 10^9 (10亿) 条记录。

AVL

AVL 是二叉平衡查找树，查找，插入，删除的平均时间复杂度都为 O(log2(n))。

mysql 随笔

mysql 中 myisam 与 innode 的区别

  1. innodb 支持事务，MyISAM 不支持
  2. innodb 支持行级锁，MyISAM 支持表级锁
  3. Innode 支持 MVCC，MyISAM 不支持
  4. Innode 支持外键，MyISAM 不支持
  5. Innode 不支持全文索引，MyISAM 支持

MVCC 与 乐观锁

多个用户/线程/进程对数据库进行操作时都会出现三种冲入情况

  1. 读 - 读：没啥冲突
  2. 读 - 写：有隔离性问题，脏读（读到未提交数据），幻读（读已提交 insert），不可重复读 （读已提交 update delete）
  3. 写 - 写：丢失更新

MVCC 是用来解决 读写 问题的一种手段。为每个写操作保存一个版本。读操作读到最新的版本数据。

乐观锁是写写 问题解决手段，在 commit 数据之前看看有没有新 commit 已经抢先提交，若是有则放弃本次 commit。

Mysql 的事务控制

Mysql 有四种级别的事务隔离

READ-UNCOMMITTED

读未提交，一般不用。不怎么加锁。以下读写问题都会有

  • 脏读
时间点 用户A 用户B
1 读 X 表 Y 行数据
2 X 表 Y 行 update 一条数据
3 rollback
4 读 X 表 Y 行数据

1、4 两次 A 用户读到的数据不一致

  • 幻读
时间点 用户A 用户B
1 读 count(X) 表的数据
2 X 表 insert 或 delete 一条数据
3 commit
4 读 count(X) 行的数据

1、4 两次 A 用户读到的数据 count num 不一致

  • 不可重复读
时间点 用户A 用户B
1 读 X 表 Y 行数据
2 X 表 Y 行 update 一条数据
3 commit
4 读 X 表 Y 行数据

1、4 两次 A 用户读到的数据不一致

READ-COMMITED

读已提交，读数据不加锁。修改、写、删除数据加锁。所以会遇到和脏读和不可重复读。

REPEATABLE-READ

可重复读，通过 MVCC 版本控制解决了 不可重复读 的问题。

幻读怎么解决呢？当前事务隔离级别采用了 Next-Key 锁。

  • Next-Key

next-key 锁是行锁 与 GAP 锁 的合并，行锁不用说了，GAP 锁又名 GAP 间隙锁。

  • GAP 锁

GAP 能锁住表级的新增。在表有索引的情况下锁住扫描过的数据。如 where a > 2 and a < 8， 就锁住[2, 8]区间不给 insert 新数据

在这个级别的事务隔离上，可以对 幻读 有一定层度的预防

详情可以看看这篇个详细的 blog :博客 mysql 加锁分析

SERIALIZABLE

完全的锁住

读未提交 V V V
读已提交 X V V
可重复读 X X V
序列化 X X X

Posted on 2019-12-02 Edited on 2019-12-03

用户与组的概念

  1. linux 多用户，多任务的特性

Linux 是一个真实的、完整的多用户多任务操作系统。可以创建多个用户、每个用户在同一时间内可以进行自己的操作完成自己的任务。

用户之间通过权限管理，用户通过不同权限完成不同的任务。实现了多用户多任务的运行机制。

  1. linux 下用户角色

在 linux 下用户根据角色定义，具体分为三种角色：

  • 超级管理员：拥有对系统的最高管理权限，默认为 root 用户
  • 普通管理员：只能对自己目录下的文件进行访问与修改
  • 虚拟用户：不能登录系统，用来运行 web 程序

nginx、noboby 用户在 linux 里面就是一个不能用账号登录的虚拟用户。为了防止程序出现安全问题。不会被黑客获取 root 权限。

  1. linux 用户和组的概念

使用一种系统资源时，须向系统申请一个用户。一个用户组是权限的集合。将用户分组是 linux 系统对用户进行管理及控制方文权限的一种手段。通过定义用户组很大程度上简化而来管理工作。

  1. 用户和组的关系

一个用户组可以拥有多个用户
一个用户可以存在多个用户组中

mongodb 随笔

Posted on 2019-12-01 Edited on 2019-12-05 In 数据库

Mongodb 概述

mongodb 属于菲关系型数据库，基于键值对存储。

  • 非关系型数据库优势
  1. 性能

nosql 不需要经过 sql 层的解析。性能较高

  1. 可拓展性

数据直接没有耦合性所以容易水平拓展

事务

事务的概念 ACID

原子性

mongo 支持单文档的原子性，就是说。更新一个 doc 时操作是原子性的

关系型数据库的则可以多行更新都为一个原子性操作。

一致性

一般关系型数据库一致性主要在单机上，mongo 多为分布式节点。一致性的讨论在各节点的数据是否一致。
mongo 可以根据业务选择强一致性 或 或弱一致性

  • read-concern 读一致性

read-concern 有好几种的级别

  1. local: 读取到任意数据（默认）

  2. available

跟 local 差不多，在分片的时候才有区别

  1. majority: 读取到大多数节点的数据(WiredTiger need)
    开启参数：replication.enableMajorityReadConcern

原理

majority 模式会起一个独立的线程 snapshot，周期性对当前的数据集进行 snapshot。并记录 snapshot 时最新的 oplog 时间戳。
当确认「oplog 已经同步到大多数 node」就把 state 更新为 commited

最新的 OPLOG 时间戳 SNAPSHOT 状态
t0 snapshot0 commited
t1 snapshot1 uncommited

如何知道oplog同步到了大多数 node

(1) secondary 节点会在 oplog 发生变化时通知 primary
(2) secondary 节点会在心跳检测的时候通知 primary 最新 oplog
(3) secondary 节点在拉取 oplog 的时候，会将 primary 上 snapshot 信息上的最新 OPLOG时间戳 同步到自身的 snapshot 信息上

  1. linearizable

在 majority 基础之上的一个更加安全的方案，就算检测到数据已经处于 majority 的状态，要检查这个 majority 里的节点是否失连。

  • write-concert 写一致性

跟读一致性差不多，写一致性是执行写操作的时候进行的，若没有达到预先定好的安全要求 response 是不会返回的。

有三个参数控制 write-concert

w: 确认写操作已经传播到了多少个 mongo 实例里
j: 确认写操作写到了 journal 日志里
witmeout: 防止达不到安全写请求无限阻塞的超时时间

  1. w (demand secondary num)

可以自己定义 demand secondary 数，比如 w = 2 时，要求主节点与一个备节点都要写入才返回写 response，在 w 也可以设置 majority。

  1. j

说 journal 之前先说下 snapshots 和 checkpoint 和 journal 的关系。

WiredTiger 学 Innodb 也搞 MVCC 版本控制。

journal 的创建、删除: 在设置 j = 1 时会在每次写操作后把操作日志写到 journal。在 下次生成 checkpoint journal 会清空。

checkpoint 的创建: 每隔 60 秒 或 日志文件达到 2GB 时(mongo3.6 后取消 2GB)，把内存中 snapshot 的文件同步到磁盘，并创建 checkpoint。

  1. oplog 与 majority 的区别

majority

journal 是数据库引擎层的概念，是持久化数据的一种辅助机制。checkpoint 的时间间隔中出现宕机，两次chenckpoint 间的数据无法恢复。

在开启 journal 后，每次写都会生成一个操作记录，在宕机后可以先用 checkpoint 恢复数据，再重放 journal 内的数据来恢复数据。

oplog

oplog 是在副本集的概念，一般用来同步 primary 与 secondary 的数据。

一次原子性写操作流程

修改数据、修改索引、生成 oplog 这三步，一步出问题则都不成功。这三步过后，如果设置了 journal 参数，则把这三步操作写入 journal 里。

  • read-reference 偏好读

一个副本集有一个 primary 与多个 secondary 组成。primary 负责写操作，数据一定是最新的。
secondary 通过 oplog 来同步写操作。数据有一定的延迟，可以在 secondary 节点查询减轻集群压力。

primary: 默认模式，一切读操作都路由到 primary 节点。
primaryPreferred: 正常的情况下路由到 primary 节点，只有当 primary 不可用时路由到 secondary 节点。
secondary: 一切读操作都路由到 secondary 节点。
secondaryPreferred: 正常情况下路由到 secondary 节点，只有当 secondary 不可用时路由到 primary 节点。
nearest: 一视同仁，从最小网络延迟节点读取。

隔离性

持久性

在 ViredTiger 引擎里，一个写操作写进了日志里就是持久的。在副本集中，一个写操作写进了大多数 mongo 节点的日志里就是持久的。

cache
